# 0. プロジェクト背景と絶対条件

## 背景
- サイト: https://www.boatship.jp/ (GitHub Pagesでホスティングされた静的サイト)
- リポジトリ: Boatshipxxx配下(サイト本体リポジトリ + kpi-tracker リポジトリ)
- Notes記事データ: kpi-tracker/notes/notes.js に `NOTES` 配列として格納。各記事は `id, num, category, title, date, readTime, theme, image, evidence, kpi{views, readRate, reactions, recordedAt}, tuningMemo, excerpt, body` を持つ
- Magazine記事: 現在 `article.html?id=01` のクエリパラメータ形式で表示

## 最終ゴール(2027年3月まで)
- 月間10,000アクセス
- 問い合わせメール50件 + 海外企業・エージェントからの問い合わせ30件
- 全記事の読了率を計測し、平均50%以上を目指す

## 絶対条件(全タスク共通)
1. **GitHub Pagesで動く静的構成を維持する**。サーバーサイド処理は使わない。ビルドが必要な場合はNode.jsスクリプト+GitHub Actionsで完結させる
2. **デザインの雰囲気(既存のトンマナ、DotGothic16等のタイポグラフィ、ブルータリズム的な意匠)を壊さない**。既存CSSを流用し、新規ページも同じヘッダー/フッターを使う
3. 各タスク完了時に必ず「変更ファイル一覧」「動作確認方法」「残作業(人間がやること)」を報告する
4. 破壊的変更(URL変更等)の前に必ずブランチを切り、mainへの直接pushはしない。PR単位で作業する
5. 不明点は推測で進めず、作業を止めて質問する。ただし1つのPRに関する質問はまとめて聞く
6. コミットメッセージは `feat:` `fix:` `content:` `chore:` プレフィックスで統一

# フェーズ1: 計測基盤(最優先・他タスクの前提)

## Task 1-1: 読了率計測スクリプトの実装
**目的**: 「読了率を大事にする」戦略の根幹。GA4に記事別の読了データを送る。

**仕様**:
- 全記事ページ(Notes/Magazine)に読み込む共通JS `assets/js/read-tracking.js` を新規作成
- 以下のGA4カスタムイベントを実装:
  - `scroll_depth` : 25 / 50 / 75 / 90% 到達時に発火。パラメータ: `article_id`, `percent`
  - `read_complete` : 「本文末要素記事フッター直前に設置する `#article-end` センチネル)がIntersectionObserverで可視化」かつ「滜在時間 >= readTime(分) × 60 × 0.5 秒」の両条件を満たしたとき1回だけ発火。パラメータ: `article_id`, `read_time_sec`
- GA4測定IDは `<head>` 内の gtag スニペットから参照する。**測定IDはプレースホルダ `G-XXXXXXXXXX` で実装し、人間が差し替える箈所としてREADMEに明記**
- SPAではないのでページロード基準でよい。ただし同一セッション内の重複発火はにsessionStorageで防止

**受け入れ条件**:
- ローカルで�いてスクロールするとconsoleにイベントログが出るデバッグモードがある(`?debug=1`)
- 全記事テンプレートにセンチネル要素とスクリプト読み込みが入っている

**想定工数: Claude Code作業 2〜3h / 人間レビュー 0.5h**

## Task 1-2: sitemap.xml と robots.txt の自動生成
**仕様**:
- `scripts/generate-sitemap.js` を作成。notes.jsとmagazineの記事データを読み、全ページのURLを含むsitemap.xmlをルートに出力
- robots.txt にsitemapの場所を記載
- GitHub Actions(`.github/workflows/build.yml`)でpush時に自動実行(フェーズ2のビルドと統合してよい)

**想定工数: 1〜1.5h / レビュー 0.5h**

# フェーズ2: SEO構造改修(最重要タスク)

## Task 2-1: 記事の静的HTML生成システムの構築
**目的**: `article.html?id=01` 形式と、JSでのクライアントサイド描画がSEOの最大のボトルネック。ビルド時に記事ごとの静的HTMLを生成する仕絸みに移行する。

**仕様**:
- `scripts/build-articles.js` を新規作成。以下を実行:
  1. notes.js の `NOTES` 配列と、magazine記事データを読み込む
  2. 記事ごとに `/notes/{slug}/index.html`、`/magazine/{slug}/index.html` を生成
  3. slugは英数字ケベブケース。既存記事のslug対応表を最初に提示し、**人間の承認を得てから**確定する(例: n01 → `equal-exchange-compensation`、magazine#01 → `shake-shack-japan-launch`)
- 生成HTMLの必須要素:
  - 記事固有の `<title>`(「記事タイトル | BOATship」形式、35文字目安)
  - `<meta name="description">`(excerptを流用、120文字以内に調整)
  - OGP一式(og:title / og:description / og:image / og:url / twitter:card)
  - canonical タグ
  - JSON-LD(`Article` スキーマ: headline, datePublished, author=Organization BOATship, image)
  - パンくずリスト(JSON-LD `BreadcrumbList` 併記)
  - 既存デザインのヘッダー/フッター/CSSを共通テンプレート(`templates/article.html`)として切り出して使用
- 旧URL対応: `article.html` は残し、`?id=XX` を検知したら対応する新URLへ `<meta http-equiv="refresh">` + JSリダイレクト + canonical で誘導
- 一覧ページ(/magazine/, /notes/)のリンクを新URLに張り替え
- GitHub Actionsでpush時に自動ビルド→gh-pagesへデプロイ(既存のデプロイ方式を先に調査し、それに合わせる)

**受け入れ条件**:
- 全記事が固有URLでアクセス可能
- Googleリッチリザルトテスト相当の構造化データ検証(ローカルでschema検証ライブラリを使ってよい)にパスする
- Lighthouse SEOスコア 95以下(任意の記事1本で確認)

**想定工数: 6〜8h / レビュー 1.5h(slug承認含む)**

## Task 2-2: 内部リンクとトピッククラスター化
**仕様**:
- notes.jsの各記事オブジェクトに `related: ["n05", "n07"]` フィールドを追加し、ビルド時に記事末へ「関連Note」カード2件を出力
- 関連付けの初期値: theme一致を優先し、あなたが本文内容を読んで&妷当な組み合わせを提案→人間承認後に確定
- 本文中の関連概念(例: n07本文の「組織同一化」→n05)にテキスト内部リンクを1記事あたり2〜3本追加。**本文の文章自体は改変しない**(リンク化のみ)

**想定工数: 2h / レビュー 0.5h**
